Rancher 容器 Pipeline CICD DevOps

为您奉上Rancher Pipeline第一时间的尝鲜体验！

Posted by Andrew Long/龙淼 on 2018-01-01

前言：
时至今日，相信大家对于来自硅谷的Rancher Labs已经不那么陌生了。我没有记错的话，Rancher应该是2015年下半年早些的时候进入中国容器技术爱好者的视野（那时国内就压根还没有“容器用户”）。Rancher主打的“开箱即用”风格的容器云产品，可快速构建和简易管理容器化的基础架构，进而提供CaaS的能力（Container as a Service）。在那个国内容器还是蛮荒的年代，Rancher的产品是足够闪耀的，所以圈粉无数。随后的几年，容器领域正可谓风云变化、大浪淘沙，容器编排领域之争现也已尘埃落定。凭借在诞生基因、开放性、可扩展性、可靠性等方面的均衡表现，Kubernetes成为最后的赢家，实属当之无愧。但笔者认为，Rancher在容器领域的技术贡献、容器在中国的推广和普及，发挥了相当积极的作用。

当今企业“数字化转型”是一个极其热门的话题，在以快为先的时代，产品可快速迭代的重要性不言而喻。CI/CD（持续集成与持续交付）敏捷、稳定、可靠的特性，越来越被企业所青睐与需要。然而真正实现CI／CD却并非易事，真pipeline搭建工作复杂，平滑升级难以保障，服务宕机难以避免等。来自硅谷的企业级容器管理平台提供商Rancher Labs，近日正式发布开源的、可与Rancher企业级容器管理平台集成的Rancher Pipeline。Rancher Pipeline会秉承Rancher一如既往的风格，给我们带来极简、易用的操作体验吗？功能方面是否全面？与传统的gitlab、jenkins可构建的pipeline又有何种不同？本文接下来的内容，就向大家奉上Rancher Pipeline第一时间的尝鲜体验！

Rancher官网：https://rancher.com
Rancher Pipeline项目地址:https://github.com/rancher/pipeline
Examples of Rancher Pipeline:https://github.com/biblesyme/pipeline-example （本文使用的Demo）
Rancher Pipeline Reference Guide：https://github.com/rancher/pipeline/blob/master/docs/README.md

Rancher Pipeline是由Rancher的应用商店(Catalog，community-catalog)服务提供，可非常简易、快速地在您的Rancher集群环境当中完成Rancher Pipeline的初始化安装；
Rancher Pipeline目前只能在Rancher 1.6版本可用，Rancher 2.0暂时还不支持。因为Rancher 2.0产品编排框架全面拥抱Kubernetes，已经抛弃了此前的自有编排框架Cattle，在应用商店（Catalog）这个服务层面会对应不同的技术实现以及开发工作量。熟悉Rancher老版本（即1.6之前）的玩家都知道，其应用商店上架主要是编写两个编排文件，docker-compose.yml和rancher-compose.yml。那么Rancher 2.0，应用包管理已经交由了现在Kubernetes生态当中流行的技术实现–HELM（源于容器创业公司Deis，该公司于2017年上半年被微软收购），所以也理所当然就有着不一样的编排文件书写规范。笔者认为，Rancher Pipeline集成进Rancher 2.0应该也不会太遥远。

2. 容器集群搭建

2.1 Rancher Server的安装

Rancher Server 1.6的安装指导详见：http://rancher.com/docs/rancher/latest/en/quick-start-guide/

因为Rancher Server服务是以容器方式来running的，所以它的安装非常简单，一条命令就可以拉起来。后面引用的镜像rancher/server，目前来看有三个tag，latest和stable是对应1.6的开发版和稳定版（开发版镜像超过1GB，建议使用阿里的镜像加速；生产部署，官方推荐使用稳定版），preview则是对应Rancher 2.0。所以，由此可见Rancher 2.0的安装还是沿袭了Rancher自有的极简风格，这点不得不点赞。只是稍加注意的是，2.0的Rancher Server，建议部署在内存大于4GB的物理机或者虚拟机上，而1.0是1GB。

当Rancher Server容器起来之后，我们便可以通过http://10.0.0.200:8080 来访问它的UI，如下：

我们可以发现，Rancher Server的UI简洁、美观，并支持在中文、英语、日语、法语等十种语言之间自由切换设置。关于Rancher的日常管理、维护，除了UI，还有另外两种途径可供选择：Rancher CLI(beta)、RESTful API。

2.2 添加主机

将10.0.0.201、10.0.0.202两台虚机添加至Rancher Server，由Rancher Server统一纳管，形成一个小规模的容器集群。此步骤，按照基础架构/主机/添加主机的UI提示进行相关操作即可。

两台虚拟机添加至Rancher Server之后，除了rancher agent，还会生成很多其他附属的、用于构建Rancher容器集群主机侧的关键基础设施容器，例如healthcheck、ipsec、network-services、scheduler，从其命名上我们就可以很容易地理解到它们的用途。rancher agent镜像只有200+MB，所以主机列表应该很快就可以看到被添加进来的主机。但是，基础设施类的容器数目比较多，总体镜像体积不小，它们变成正常的、Running状态，可能需要花上一些时间（考验网速的时候到了:)）。在基础设施类的容器没完全拉起来之前，严格意义上来说，集群是不完整的、不可用的。

2.3 安装Docker Registry v2

在10.0.0.200上额外安装Docker Registry v2（镜像库，可以使用类似 VMware Harbor、Youruncloud AppHouse、Nexus代替），然后在10.0.0.201和10.0.0.202上的docker daemon配置insecure-registries选项。根据笔者的环境，修改/etc/docker/daemon.json文件添加insecure-registries选项、再重启docker。当然，不要忘记在10.0.0.201和10.0.0.202上做一下push镜像的测试，确认其有效性。

在Rancher UI上也需要添加我们刚新建的镜像库。Rancher可以把 DockerHub、Quay.io 或者任何私有镜像库的访问凭据添加进来，来访问私有镜像。在镜像库被添加进来之后，Rancher将能在启动服务或者容器的时候使用这些私有镜像库。镜像名称的语法是与我们在docker run 命令中是相同的。

3. 部署Rancher Pipeline

3.1 在应用商店中搜索”pipeline”

3.2 部署Rancher Pipeline

找到Rancher pipeline之后，点击查看详情。在新弹出的展示页面中，可保持默认配置，直接点击“启动”按钮，Rancher Server则将自动化新增部署Pipeline组件。在点击“启动”按钮之前，点击“预览”，通过docker-compose.yml内容不难看出，Rancher pipeline服务是主要由四部分组成，jenkins-master、jenkins-slave、pipeline-server、pipeline-ui。

需要注意的是，Rancher pipeline相关的服务容器也是属于关键基础设施容器。

4. 配置Rancher Pipeline

4.1 添加流水线，包括添加Repo、添加Github的授权

当第一次添加流水线时，Rancher会提醒您添加代码库、以及代码仓库的授权。

根据页面提示，一步步完成Repo、Github的授权的添加。

选择提前Fork自官方的github项目pipeline-example，保持默认设置中的master分支、勾选Webhook，点击Add按钮。

Rancher Pipeline支持通过导入pipeline.yml文件快速创建流水线，而编写pipeline.yml有一定的语法规范，笔者没有做过多了解，应该和jenkins pipeline文件采用的groovy语法差不多。

4.2 配置流水线

随后，我们可以按照https://github.com/rancher/pipeline 项目README.md页面中的Quick Start Guide，对流水线的Stage和步骤进行配置。

Stage，中文为“阶段”，意指流水线中类似源码、测试、打包等这一个个连串的阶段过程，上一个Stage结果Success流水线才会推进到下一个Stage，否则终止结束；而每一个Stage，又可以定义很多并行或者串行的步骤（例如，测试这个Stage，可以设计由单元测试、静态扫描、功能测试、性能测试、UI测试等步骤来任意组合），每一个步骤可以指定Shell脚本、Entrypoint，并引用自定义变量；如果某Stage有一个步骤运行结果为Fail，那么该Stage就为Fail，整个流水线也会终止结束。
